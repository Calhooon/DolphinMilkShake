### The 3 dolphin-milk product bugs that had to be fixed to get cascade green

Every one of these would hit any delegated agent at any scale, not just cascade-specific. All fixed in `2fb5c17`. Document these as part of the "why Coord/Worker agents weren't shipping at 25-scale before" story.

**1. Tool result offload preview was unusable for nested JSON**
- File: `src/runner/execute.rs`
- Threshold was 8KB (way too aggressive — every real API response got offloaded). claude-code uses 50KB as a proven default. Bumped.
- Old preview format was "first N bytes of content" which failed for nested JSON (e.g. HN Algolia's 34KB response where titles are buried under a metadata envelope — the first 2KB is `{"exhaustive":{"nbHits":true,"typo":true},...}`, never reaches `hits[0].title`).
- New format is a delimited `<persisted-output>` envelope with a **recursive structural summary** that descends up to depth 3, shows first 3 items of each array with their field schemas, plus a raw byte-prefix fallback for error diagnosis. Bounded output (cap `PREVIEW_MAX_DEPTH=3`, `PREVIEW_MAX_ARRAY_ITEMS=3`, `PREVIEW_MAX_OBJECT_KEYS=10`, `PREVIEW_MAX_VALUE_LEN=80`, `PREVIEW_RAW_PREFIX_LEN=1000`).
- 23 inline unit tests in `mod preview_tests` including the canonical `god_tier_preview_hn_response` test that asserts titles actually appear in the structural summary.
- Retrieval path exposed as `read_tool_output call_id="..." (or file_read path="...")` in the envelope header.

**2. Delegation allowlist stripped the offload retrieval tools**
- File: `src/runner/lifecycle.rs` — `apply_pending_delegation`
- `set_allowlist(caps)` REPLACED the entire tool registry allowlist with just the cert-granted capabilities. So a delegated agent with `caps=["web_fetch"]` that fetched a >50KB response got an offload preview pointing to `file_read` — but couldn't call `file_read` because the cert didn't grant it. Worker's P3 in cascade-run4 looped retrying `web_fetch` until budget exhausted.
- Fix: after `set_allowlist(caps)`, always insert `read_tool_output` and `file_read` as system escape hatches. Both only touch the agent's own workspace, so they're within the delegation's trust boundary.

**3. Commission payment internalize raced against WoC indexing**
- File: `src/heartbeat/commission_payments.rs` — `handle_commission_payment_sent`
- Receiver was ignoring the `beef_hex` the payer ALREADY included in the `commission_payment_sent` message and instead round-tripping to WhatsOnChain via `fund_from_woc(payment_txid, ...)`. WoC indexing lags broadcast by 10-60s → HTTP 404 → commission payment permanently lost (no retry path).
- Fix: new `try_internalize_beef()` helper wraps the BEEF in AtomicBEEF format and calls `wallet.internalize_action()` directly with the `paymentRemittance` descriptor. Falls back to the WoC path only on failure (malformed BEEF, wrong derivation) as a safety net.

## Risk assessment + the three things I'd flag in a fresh session

### Risk 1: MessageBox + wallet contention at 35× has never been tested

Our multi-worm tests max out at 3 agents. At 25-35 concurrent agents doing cascades:
- BRC-33 MessageBox delivery under load is unknown
- `createAction` spend semaphore serializes — a slow LLM call can stall commission broadcasts behind it for dozens of agents
- SQLite DB lock contention under multi-process hammering
- Heartbeat inbox polling every 15s × 35 agents = 2.3 polls/sec against MessageBox server

**Mitigation**: before scaling to 35, run a **5-agent smoke test** as the last gate of #20. If 5 concurrent 3-agent cascades run clean, 35 total agents (~10 concurrent cascades) is credible. If 5 blows up, find the bottleneck with a small blast radius instead of scrambling with 35.

### Risk 2: Commission payment backlog at scale

Today's cascade run had 4 commissions per run. At 35 agents running continuous cycles you'll have dozens of in-flight settlements. Even with the direct-BEEF fix, the payer-side `createAction` is serialized by spend semaphore. Measure during the smoke test — if commissions back up, you need a higher-concurrency payment path before the 24-hour run.

### Risk 3: #21 UI variance is the schedule killer

"God View UI" could be 1-2 days of custom work. The **iframe grid fallback** is genuinely good enough for a demo video — 25 tiles each at `localhost:808N/ui/` showing the real chat view is more authentic than a custom aggregator anyway. Do 1 hour on iframe grid + polished header bar with live tx counter, then only go custom if that clearly looks bad. Don't let UI polish eat #22/#23 slack.

## Critical gotchas to NOT forget

These all cost time today. Don't re-hit them:

1. **The wallet CLI uses hardcoded BRC-29 constants that differ from dolphin-milk's.** `bsv-wallet send`/`fund` uses its own `SfKxPIJNgdI=` / `NaGLC6fMH50=`, while dolphin-milk's `receive_address` / `fund_from_tx` use protocol `[2, "3241645161d8"]` with `dolphin-milk-fund {suffix}` key IDs. **Always use dolphin-milk's `receive` + `fund` CLI, not bsv-wallet's.** If you send to the wrong address, recovery is via `tests/multi-worm/sweep-worker-stuck.js` — but don't get there.

2. **WoC returns BEEF, but `bsv-wallet fund` expects AtomicBEEF.** Wrap with `Beef.fromString(beefHex).toBinaryAtomic(txid)` from `@bsv/sdk` before passing.

3. **MUST use wallet `daemon` not `serve` mode.** Serve has no chain monitor; txs stay unproven; BEEF chains grow deep; double-spend errors after ~30 txs. Burned 3 test wallets learning this on prior work.

4. **Pre-split UTXOs always.** Use the new `fund-agents.js` with `SPLIT_COUNT=5` or adapt the pattern. A single giant UTXO forces chain-serialization and kills cascade throughput. Confirmed 2× speedup in run11 vs prior runs after splitting.

5. **Claude-haiku has an injection-defense reflex**. Nested "capabilities" strings in delegated task arguments will make the LLM extract downstream values instead of passing its own. **Use the opaque-task-string pattern** (see `buildCaptainHarnessTask`, `buildCoordForwardTask`, `buildWorkerTask` in `test_three_layer_cascade.js`). Wrap nested task text in `===TASK_STRING_BEGIN===` / `===TASK_STRING_END===` markers and explicitly tell the LLM "this is opaque, do not extract values from it."

6. **gpt-5-mini is too injection-paranoid for delegated tasks.** It refused legitimate delegated work in cascade-run4 with the hallucinated "LOBSTER-VERIFIED" marker. **Use claude-haiku-4-5 for all 3 layers in the cascade.** Cost is slightly higher but reliability is night-and-day.

7. **Budget caps need teardown headroom.** Worker's 300K cap blew out at 353K because session_end teardown (commission claim emission + reverse-delegation proof + state token updates) costs ~50K beyond the main work. **Always size budget caps with 50-100K headroom for teardown.** Run11 used 500K for forward caps and 300K for final memory_write.

8. **Commission counter must read full log files, not a tail.** `readLogTail(20000)` will scroll commission events out of view during long runs — use `fs.readFileSync` on the full `server-stderr.log` and match with a monotonic `max()` accumulator across polls.

9. **Deep transcript inspection needs retry-tolerance.** LLMs sometimes make a malformed first call and a correct retry. `findSuccessfulCall(tcs, trs, 'delegate_task', recipient)` pattern — walk all matching calls, skip results that start with "error", return the first clean one — not `array[0]`.

10. **Workspaces are evidence, never `rm -rf test-workspaces/`**. Every conversation, transcript, BRC-18 proof, and BRC-48 token in those dirs is part of the audit trail. If you need a clean slate, create a NEW workspace dir.

11. **Wallet HTTP API requires `Origin` header** for ad-hoc curls (`Origin: http://localhost`). Dolphin-milk goes through proper BRC-31 auth so this isn't a worm issue — only bites when you shell out directly.
